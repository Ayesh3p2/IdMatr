IDMatr Security Remediation Report

Code and configuration remediation summary for the IDMatr multi-tenant SaaS platform following the enterprise compliance audit.

This report reflects repository remediation work completed on March 14, 2026. It is a code-level remediation report, not a substitute for a live deployment verification or external certification audit.

Repository: IDMatr Report Date: March 14, 2026 Mode: Autonomous Remediation Scope: Backend, Gateway, Control Plane, Frontend, Docker, CI

Executive Summary

The remediation pass materially improved the highest-risk findings from the original audit. The codebase now enforces secret presence at startup, removes the X-Tenant-ID tenant override path, shifts browser auth away from token storage in localStorage, introduces one-time onboarding links in place of plaintext temporary passwords, adds TOTP MFA support for privileged users, encrypts sensitive settings and integration payloads at rest, and hardens NATS with private-only broker access plus per-service credentials.

Tenant scoping was strengthened in the graph and policy services, the control plane now records tamper-evident audit metadata, and a baseline security CI workflow was added for SAST, secret scanning, dependency review, and tests.

Readiness conclusion: substantially improved, but not yet certification-ready. The remaining blocker is validation confidence rather than a lack of core remediation code. The current execution environment would not successfully run the local node/npm/prisma verification commands, so build/test/runtime proof is incomplete. A controlled validation cycle is still required before production sign-off.

Completed Remediation by Phase

Phase Status Remediation Outcome
1. Internal Trust Boundary Implemented NATS public exposure removed from compose, broker config added under deploy/nats/nats-server.conf, and per-service NATS usernames/passwords were introduced. Gateway and services now require authenticated NATS connection options.
2. Tenant Isolation Implemented X-Tenant-ID override support was removed from the gateway middleware. Tenant context is derived only from the authenticated JWT for protected tenant routes.
3. Graph Service Security Implemented Graph queries now require tenantId and constrain users, roles, permissions, applications, and path traversals to the same tenant boundary.
4. Secret Management Implemented Fallback JWT and internal secrets were removed. Startup now fails if critical secrets such as JWT_SECRET, CONTROL_PLANE_JWT_SECRET, INTERNAL_API_SECRET, or DATA_ENCRYPTION_KEY are missing.
5. Onboarding Security Implemented Temporary password generation, UI display, and logging were replaced with one-time onboarding links backed by hashed onboarding tokens, 15-minute expiry, and single-use invalidation.
6. RBAC Implemented JWTs now carry normalized roles such as platform_operator, tenant_admin, tenant_user, and viewer. Hardcoded tenant admin role issuance was removed. Control-plane tenant APIs are restricted to platform operators.
7. Session Security Implemented Browser auth flows were migrated from token persistence in localStorage to HTTP-only session cookies for both tenant and control-plane flows.
8. MFA Partially Implemented TOTP setup, enable, disable, and enforcement were added for platform operators and tenant administrators. Backend coverage is present, but UI enrollment workflows are still limited and need runtime verification.
9. Secret Encryption Implemented Envelope encryption was introduced for sensitive stored values. Control-plane settings and tenant integration configuration now encrypt sensitive data before persistence.
10. Rate Limiting Implemented Stricter in-process rate limiting was added for gateway auth routes and control-plane auth routes.
11. Audit Logging Partially Implemented Control-plane auth and administrative actions now write chained audit hashes to support tamper evidence. Full immutable coverage across all microservices still requires broader adoption.
12. Infrastructure Hardening Implemented Internal services are no longer broadly host-exposed. NATS is private-only and internal databases/broker/control-plane services are loopback-bound or internal-only in compose.
13. Security CI Implemented A GitHub Actions workflow now runs gitleaks, Semgrep, dependency review commands, Go vuln scanning, and workspace tests.
14. Privacy Controls Partially Implemented Tenant export and hard-delete paths exist and retention metadata fields were added. Full retention enforcement automation and user-facing DSAR workflow coverage still need follow-up.
15. Final Validation Incomplete Final runtime validation was blocked by local execution issues with node, npm, and prisma commands in this environment. Static remediation review was completed, but successful build/test evidence is still required.

Key Repository Changes

Gateway and Auth

  • Cookie-based auth added for tenant and operator sessions.
  • Tenant login now supports disambiguation via tenantSlug and privileged TOTP verification.
  • Gateway MFA endpoints added for tenant administrators.
  • JWT extraction now supports secure cookies plus bearer tokens for compatibility.

Control Plane

  • One-time onboarding token model added to the Prisma schema.
  • Operator and tenant MFA secrets are stored encrypted.
  • Tamper-evident audit hashing introduced for control-plane events.
  • Tenant export endpoint added for privacy operations.

Data Protection

  • Envelope encryption added for SMTP secrets, API tokens, and integration config.
  • Raw integration credential blobs are no longer returned from tenant detail APIs.
  • Temporary password disclosure removed from UI, email logs, and API responses.

Infrastructure

  • NATS hardening config added with per-service credentials and subject restrictions.
  • Docker compose exposure reduced for databases, Redis, Neo4j, NATS, and control plane.
  • Environment template updated with required secrets and NATS service passwords.

Production Decision

Decision: not yet approved for enterprise production deployment.

Once the runtime verification pass succeeds and the residual items above are closed or formally accepted, IDMatr should be re-scored for final production and certification-readiness sign-off.
